How to read the resource timeseries

  1. X axis — elapsed seconds. t = 0 is when the benchmark harness started its timed loop (right after artifact prep). Not absolute clock time. The right edge is roughly total benchmark duration.
  2. Coloured shaded bands — each band is one full test mode window (passthrough → compression → encryption → compression+encryption). Left-to-right you see the entire suite in sequence — not a single mode in isolation.
  3. Vertical dashed lines — a timed phase began: PUT, cold GET, or warm GET for the labeled mode (passthrough → compression → encryption → compression+encryption). Alternate amber / blue for readability.
  4. Hover any plotted point — tooltip shows measured values; footer lists phase markers near that time and the most recent phase that started before this t.
  5. CPU — Docker-reported CPU% for the dgp-bench container.
  6. RAM — proxy process RSS (Prometheus/health) vs Docker cgroup memory (parsed from MemUsage).
  7. Disk — du -sb trees for plain vs encrypted backend roots; expect jumps when modes switch buckets.

Sampling interval: 2.0s between resource snapshots (whole run).

Conclusions

Compression reduced stored bytes. Approximately 150.00 MB saved vs logical uploads (75.00% of logical bytes).

PUT throughput drops from 80.0 MB/s (passthrough) to 40.0 MB/s (compression) — delta encode is CPU-heavy vs streaming writes.

The encrypted backend bucket measured 55.0 MB/s PUT vs passthrough — AES-GCM adds measurable CPU cost while preserving logical bytes on read.
