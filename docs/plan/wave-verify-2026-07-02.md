## 1. [HIGH] Finding #4 fix regresses on the SECOND verify: parity-cache hit skips the dest-orphan ownership HEAD-overlay, so a rule-owned S3 orphan is again misdiagnosed as ForeignOrphan
`src/replication/parity.rs` — streaming-fixes

The Wave-2 remediation of finding #4 makes the dest scan set owned_by_rule provisionally to Some(false) on a non-lite-authoritative (S3/encrypting) dest, then overlays real ownership from the HEAD burst in resolve_logical_dest (parity.rs:707-709, only on HeadOutcome::Resolved). But resolve_logical_dest consults the DB-backed parity cache FIRST (cache_get at :689, cache_hit_fresh at :694) and applies a cache hit via apply_logical(). apply_logical (parity.rs:569-580) copies sha256/size/etag/multipart_parts and NEVER touches owned_by_rule; ParityCacheEntry has no owned_by_rule field at all. On the FIRST verify the orphan is HEAD-resolved (ownership Some(true)) and its entry is written to replication_parity_objects with a stored_etag. On the SECOND verify of the same rule, the orphan's stored_etag still matches the S3 LIST etag (cache_hit_fresh true), so the HEAD burst is skipped, owned_by_rule stays the provisional Some(false), and the rule-owned orphan is analyzed via dest_owned_by_rule=Some(false) (parity.rs:470) as foreign_orphan — the exact #4 symptom the fix was meant to kill. Failure scenario: S3 dest, source key deleted after copy; operator verifies (correct: rule_owned_orphan_source_deleted), then re-verifies (the natural 'did my fix take' flow) → same orphan now reported foreign_orphans+1 with 'delete it manually' guidance and the correct RuleOwnedOrphanSourceDeleted remediation can never resurface. The new integration test test_parity_orphan_ownership_via_head_on_s3 (tests/parity_test.rs:272) does a SINGLE cold-cache verify and so does not exercise this. Fix: carry owned_by_rule in ParityCacheEntry and re-apply it in apply_logical, or exclude orphans from the cache-hit fast path on non-lite-authoritative dests.

## 2. [HIGH] Finding #5 fix is one-directional: an ENCRYPTED (or S3) replication SOURCE still compares ciphertext lite size for non-delta keys → mass false ChecksumMismatch on a converged mirror
`src/replication/parity.rs` — streaming-fixes

The Wave-2 remediation of finding #5 added dest_lite_authoritative = lite_list_carries_logical_facts(destination.bucket) and forces every dest key to be HEAD-resolved when the DEST list is untrustworthy (parity.rs:958, 979). But the SOURCE scan was NOT given the symmetric guard: line 945 still queues source keys for logical resolution only via needs_logical_resolution(engine, key, meta), which is meta.is_delta() || is_delta_eligible_key(key) (parity.rs:564-566) and does NOT consult lite_list_carries_logical_facts(&rule.source.bucket). There is no src_lite_authoritative variable anywhere. Failure scenario (the finding's explicit 'or vice versa'): replicate FROM an aes-gcm-encrypting S3 backend TO a plaintext backend. Every non-delta-eligible source key (images, media, checksum sidecars) is left with its lite CIPHERTEXT size (whole-object +28B IV+tag, or chunked header + ~per-frame overhead) from the source LIST; it is present on both sides so it survives src_needs_logical.retain (:994), never gets a source HEAD, and compare_pair (parity.rs:221-227) treats the ciphertext-vs-plaintext size delta as an authoritative ChecksumMismatch → in_sync=false with up to 100 mismatch samples on a mirror that is actually converged. Filesystem-encrypted sources escape via xattr lite listings, so the prod encrypted-filesystem fixture hides it, but an encrypted-S3 source does not. Fix: mirror the dest change — compute src_lite_authoritative and queue source keys for HEAD when !src_lite_authoritative, as done for the dest.

## 3. [HIGH] Transient HEAD failure on a dest-only orphan silently drops it → false "in_sync" verdict (parity remediation regression)
`src/replication/parity.rs` — parity-fixes

In `resolve_logical_dest` (src/replication/parity.rs:719-721) a `HeadOutcome::Unresolved` (transient throttle/5xx on the ownership HEAD) does `dest.remove(&k)`, identical to the `Gone` (raced-delete) case. The #16 rationale — "drop the key rather than emit a false verdict off an untrusted lite size" — is sound for a key present on BOTH sides (where the risk is a false ChecksumMismatch), but it is WRONG for a dest-ONLY orphan: dropping it does not avoid a false verdict, it CREATES one. On an S3 destination `lite_list_carries_logical_facts` returns false, so EVERY dest key (including every orphan) is queued into `dst_needs_logical` (line 979) and kept by the retain (line 1004, `|| !dest_lite_authoritative`), then HEAD-resolved. `head_burst` (lines 740-766) issues each HEAD with NO retry — unlike `scan_prefix`'s LIST, which retries transient errors up to `LIST_MAX_ATTEMPTS` precisely because Hetzner/S3 503-throttle long scans. So a single transient 503 on an orphan's HEAD → `Unresolved` → `dest.remove` → the orphan disappears from the `dest` map → `diff_parity` never sees it → `orphan_on_dest` under-counts it. Because an `Unresolved` outcome sets neither `truncated` nor any error (grep confirms: truncated only tracks the scan cap), `in_sync` (lines 1072-1076) can compute `true` even though a genuine orphan exists on the destination — the exact "is my mirror verified identical?" question the module was built to answer, answered wrong. Failure scenario: S3 dest under load has a foreign/rule-owned orphan `k`; parity LIST succeeds, the ownership HEAD for `k` gets one 503; `k` is dropped; audit reports orphan_on_dest=0, truncated=false, in_sync=true. Same hole hits a delta-eligible orphan on a filesystem dest (it too enters `dst_needs_logical` via `needs_logical_resolution`). Fix: for the orphan case, an `Unresolved` HEAD must NOT drop the key — keep it in `dest` (ownership falls back to the provisional `Some(false)`/`None`, yielding a ForeignOrphan/ManualReview finding) and/or mark the audit partial (`truncated=true`) so `in_sync` can't read true. Do not treat orphan resolution failure like a raced delete.

## 4. [HIGH] retain-newest collect loop ranks & deletes over a page-budget-truncated candidate set (data loss)
`src/lifecycle/worker.rs` — unverified-batch-2

run_or_preview_retain_newest (src/lifecycle/worker.rs:455-514) collects candidates with `while let Some(page_idx) = pager.begin_page()`, which returns None once pages_started >= MAX_JOB_PAGES (10,000). The loop guards ONLY against MAX_RETAIN_NEWEST_CANDIDATES (200k candidate count, :487) — it never calls `pager.truncated_by_page_budget()` after the loop. When the page budget is exhausted while the S3 listing is still truncated (e.g. a prefix with tens of millions of objects, or where most keys are filtered out by include/exclude globs so candidates accumulate slower than pages burn — page_size can be as low as 1 via batch_size, so 10,000 pages can hold far fewer than 200k candidates), the loop exits with an INCOMPLETE candidate set and proceeds straight to plan_retain_newest (:517) which ranks newest-first over ONLY the collected prefix and, in execute mode, deletes everything past `count` (:552-583). S3 LIST returns keys lexicographically; when key order and created_at order diverge (UUID-prefixed keys, or the newest backups sorting lexicographically LAST beyond page 10,000), objects that are genuinely in the true newest-N are absent from the collected set and objects that were listed but are actually NOT the oldest get deleted — deleting backups the operator intended to keep. The doc comments at worker.rs:394-398 and planner.rs:449-451 explicitly promise the design FAILS rather than rank a truncated set, but that guarantee only holds on the candidate-cap path, not the page-budget path. The age-based lifecycle path is safe (cursor-driven, preserves its token on truncation); retain-newest is set-relative and cannot resume, which is exactly why it must fail on page-budget truncation. The DGP_TEST_MAX_JOB_PAGES seam (used in tests/replication_test.rs:2346) trivially reproduces it; no test guards this path. Fix: after the collect loop, if `pager.truncated_by_page_budget()` is true, fail the run (as the candidate-cap path already does) instead of ranking/deleting.

## 5. [HIGH] Wave-4 mid-run maintenance-gate deferral (finding #12) is replication-only — lifecycle runs keep writing/deleting into a bucket being migrated
`src/lifecycle/worker.rs` — unverified-batch-2

The Wave-4 fix (commit 9fb56ba) added a mid-run MaintenanceGate check to replication: run_rule takes Option<Arc<MaintenanceGate>> and RunControl.check() returns Paused (cursor preserved) when the dest bucket becomes write-gated mid-run (src/replication/worker.rs:136,1049-1090; scheduler passes Some(gate) at src/replication/scheduler.rs:154). The lifecycle worker was NOT changed (git diff b6cb749..HEAD on src/lifecycle/worker.rs is empty). lifecycle::run_rule (src/lifecycle/worker.rs:80-88) takes no maintenance_gate, and its per-page loop (:267-273) only calls renew_run_lease — there is NO mid-run is_busy check. Lifecycle is gated against maintenance only at run START (scheduler is_busy at src/lifecycle/scheduler.rs:156), proving the gating is intended, but the mid-run window is left open. Concrete failure: a lifecycle transition run that is already in-flight (copy_object_with_retries to destination_bucket at src/lifecycle/worker.rs:668-680, optionally delete_source) — or a retain-newest/expire run deleting from rule.bucket — keeps writing/deleting into that bucket for the entire (potentially minutes-to-hours) run even after an operator starts a migrate/re-encrypt on it. During migrate's stage->copy->verify->flip, concurrent lifecycle deletes remove objects migrate already staged and lifecycle writes add objects migrate won't copy, so verify passes on a set that diverges from the post-flip target -> silent data divergence/loss. Fix: thread the MaintenanceGate into lifecycle run_rule and defer mid-run like replication's RunControl.

## 6. [MEDIUM] ⌘S is dead on the entire System settings page (nav entry lacks applyKey)
`demo/s3-browser/ui/src/components/adminNavigation.tsx` — unverified-batch-1

PREVIOUSLY-UNVERIFIED finding #1 — CONFIRMED present at HEAD.

The System nav entry (adminNavigation.tsx:214-227, path 'system') declares `dirtyKeys: ['system/listener','system/caches','system/logging','system/sync']` and `saveModel: 'review'` but has NO `applyKey`. Every other review-model page sets one (access/credentials, access/admission, storage/buckets, jobs, integrations/event-delivery).

AdminPage.tsx:115 computes `activeDirtyKey = applyKeyForPath(adminPath)`, and applyKeyForPath (AdminPage.tsx:948-949) returns `findEntry(ADMIN_IA, path)?.applyKey`. For 'system' that is `undefined`. The ⌘S handler (AdminPage.tsx:185-194) guards with `if (activeDirtyKey && requestApplyCurrent(activeDirtyKey))` and requestApplyCurrent (useDirtySection.ts:133-140) returns false immediately on a null/undefined key. So ⌘S never dispatches on the System page.

Meanwhile each System card DOES register an apply handler: SystemPanel → advancedPanels.tsx → useAdvancedSubset(..., 'system/listener'|'system/caches'|'system/logging'|'system/sync') → useSectionEditor, which calls `useApplyHandler(dirtyKey, runApply, isDirty)` (useSectionEditor.ts:263). Those handlers sit in the registry under the system/* keys but are unreachable because the page has no applyKey to dispatch to them.

Failure scenario: operator edits a field in Listener & TLS (or Caches/Logging/Sync) on /_/admin/system; the 'Review & apply' badge and the inline dirty bar appear, promising ⌘S works; operator presses ⌘S; nothing happens (on some browsers the native 'save page' dialog fires instead, since preventDefault is only called when a handler ran). The edit stays unsaved with no feedback. Note that even adding a single applyKey cannot fully fix this — the page has four independent dirty editors and the one-key-per-page dispatch model can address only one; the System page needs a different (per-visible-card) dispatch or a merged apply. The concrete regression to report is that ⌘S is currently a complete no-op on System while the UI advertises review-and-apply.
