Fingerprinting the S3 implementation behind the endpoint

How to tell MinIO from Ceph from R2 from AWS using nothing but HTTP responses — and why, for the one capability the proxy actually needs to know (conditional writes), you should probe the behavior, not the badge.

Bottom line

Fingerprint the vendor for telemetry; probe the capability for correctness. Conditional-write support is a property of vendor × version × bucket-config × fronting-proxy, not of the vendor name. The failure mode of guessing wrong is silent data loss, so the compare-and-swap decision must come from a live two-step probe that fails closed.

01 / HEADERS

Response headers that leak identity

The cheapest signal, and the least trustworthy — a reverse proxy (nginx, CloudFront, Cloudflare) in front of the origin routinely rewrites or strips Server and x-amz-*. Treat a matching Server string as a strong positive, but never treat its absence as a negative.

The captures that prove the point

Two live probes taken while writing this show exactly why the Server header alone is a trap — and how the body + secondary headers rescue the identification:

# play.min.io — a real MinIO, but fronted by nginx:
Server: nginx/1.24.0 (Ubuntu)       ← Server says nginx, not MinIO!
X-Amz-Request-Id: 18BEBD140E1DBDB4     ← 16 uppercase HEX = MinIO tell
X-Amz-Id-2: 3e996b2f…8681eac          ← 64ch lowercase hex
X-Xss-Protection: 1; mode=block       ← MinIO sets this; AWS/Ceph don't

<Error>…<BucketName>…</BucketName>
       <Region>us-east-1</Region>          ← <Region> in error body = MinIO
       <HostId>3e996b2f…8681eac</HostId>  ← HostId == x-amz-id-2 = MinIO
Takeaway

When a proxy overwrites Server, MinIO still gives itself away four other ways: hex-timestamp x-amz-request-id, 64-char-hex x-amz-id-2, X-Xss-Protection, and the <Region> element in the error body (where HostId mirrors x-amz-id-2). Fingerprinting = weighing several weak signals, not reading one header.

Two dialects. High-compatibility servers answer a never-configured sub-resource with 404 NoSuch…; leaner servers answer with 501 NotImplemented. That split is itself a fingerprint.

Hardcoded values — the deterministic tells

  • MinIO ListBuckets Owner is hardcoded on every install: ID 02d6176db174dc93cb1b899f7c6078f08654445fe8cf1b6ce98d8855f66bdbf4, DisplayName minio. Single call, deterministic — the strongest tell in the set.
  • GetBucketLocation LocationConstraint: Garage → literal garage; R2 → auto; B2 → us-west-00N; MinIO → empty by default; SeaweedFS → wrong root element <CreateBucketConfiguration>.
  • Multipart ETag: everyone uses AWS md5(md5s)-N except R2 (non-reconstructable) and B2 (SHA1-native, opaque) — both break s3cmd/s5cmd MD5 checks, a strong B2/R2 tell.
03 / CAPABILITY

The decision that actually matters: conditional writes

The proxy needs one yes/no answer — does this backend honor compare-and-swap (If-None-Match: * / If-Match: <etag> on PUT)? This is the highest-signal probe and the most dangerous one to get wrong.

⚠ The silent-ignore hazard

A naive one-shot probe — "send a conditional PUT, saw 200 → supported" — concludes supported on exactly the backends that will later eat your data. Old MinIO and SeaweedFS (on versioned/locked buckets) accept If-None-Match: *, ignore the precondition, overwrite, and return 200. The client believes it won an exclusive create; it actually clobbered another writer.

A 200 is ambiguous. Never let a lone 200 mean "supported."

04 / PROBE

A safe two-step probe that can't be fooled

Force the precondition to be the only possible reason for failure, then assert on the rejection — the one outcome a silent-ignore server can't fake.

  1. Establish existence. Plain PUT a random scratch key you own, e.g. .dgp-capprobe/<uuid>. Expect 200. The object now provably exists.
  2. The discriminating request. PUT the same key again with If-None-Match: *. This is the assertion that can only pass if the precondition was actually evaluated.
  3. Read the status — and only trust the rejection. See the outcome table below.
  4. Confirm the positive path (optional). A fresh random key + If-None-Match: * should return 200 (created). A backend that 412s a brand-new key isn't honoring semantics either — fail closed.
  5. Clean up. DELETE the scratch keys.
412CAS honored — the only success signal. AWS, R2, MinIO ≥ 2024-09-13.
200Precondition silently ignored → NOT supported. The object was just overwritten. This is the trap a one-step probe fails. Old MinIO, SeaweedFS on versioned buckets.
501 / 400Explicitly unsupported. Wasabi, B2. Safe — the server told you the truth.
500Broken implementation (s3proxy-style). Treat as unsupported.
Design rules that make the probe correct

Order is load-bearing. PUT-then-conditional-PUT and expect the 412. Expecting success proves nothing — a silent-ignore server returns 200 too. Expecting 412 on a known-existing key is the only assertion that requires the precondition to have run.

Sign with SigV4. AWS-compatible impls only evaluate conditional writes under SigV4; a SigV2/legacy-presigned request makes the header a silent no-op — a false negative and a silent-ignore source.

Probe per representative bucket, then cache. Capability flips with bucket config (versioning + object-lock changed SeaweedFS's answer). If you write to versioned buckets, probe a versioned bucket. Cache the result per destination bucket, not once per backend.

Fail closed. Any non-412 outcome → assume no CAS, fall back to the lock-based path. The costs are asymmetric: a false "supported" is silent overwrite; a false "unsupported" is a slower write. Bias hard toward unsupported.

AWS reference semantics (the contract you're testing against)

  • If-None-Match: * — no object → 200; object exists → 412 PreconditionFailed. Concurrent writers: first wins, rest get 412. A concurrent delete can surface 409 ConditionalRequestConflict (retryable).
  • If-Match: <etag> — match → 200; mismatch → 412; key deleted underneath → 404 (reupload).
  • Requires SigV4. Works on PutObject, CompleteMultipartUpload, CopyObject. If-None-Match on PUT only accepts the literal * at AWS.
05 / TOOLING

Is there a library that already does this?

Short answer: no purpose-built "which S3 vendor is this" library exists. What's out there splits into header-matchers and compatibility suites that assume you already told them the backend.

  • nuclei-templates — MinIO coverage is misconfig/exploit-oriented (default minioadmin login, console exposure), not vendor fingerprinting. Its model is "match a Server header / body regex" — exactly the brittle approach.
  • ceph/s3-tests — the canonical compat suite does not auto-detect. You hand it an endpoint + creds; gating is by pytest markers you choose. Telling signal: even the reference suite treats capability as a configured input, then verifies with real requests.
  • warp / s3-benchmark — throughput benchmarks, zero vendor logic.
  • Wappalyzer / WhatWeb — generic header/HTML matchers; object storage isn't their focus; no robust MinIO/Ceph signature set.
  • Shodan / Censys — no official filter; practitioners dork on the leaked Server / x-amz-* shapes described in §1.
  • Rust / Go capability probers — none found. The ecosystem just tries the operation via the AWS SDK. That absence is the recommendation: the field probes, it doesn't fingerprint.
06 / RANKING

Most reliable fingerprints, in order

If you build a detector, weight the signals like this — hardcoded determinism first, then the safety-relevant behavioral probe, then the shape tells.

1
MinIO

Hardcoded ListBuckets Owner ID

02d6176d…f66bdbf4 + DisplayName minio, identical on every install. One call, deterministic.

2
safety-critical

CAS PUT on an existing key

412 vs silent 200 vs 501 partitions the field and flags the dangerous silent-overwrite backends. This is the one you actually need.

3
Garage

Four independent tells

LocationConstraint garage; error body has <Region> but no ids; no SignatureDoesNotMatch; ListBuckets has no xmlns.

4
Ceph family

tx-id + missing x-amz-id-2

tx… request-id with no x-amz-id-2. Split by zone suffix: -default vanilla, -nyc3a DO, -fsn1…ceph4 + x-rgw-* Hetzner.

5
dialect

501 on GetBucketReplication

Separates Garage / R2 / B2 / Wasabi (501) from the 404-NoSuch… compat backends (AWS / MinIO / Ceph / SeaweedFS).

6
B2 · R2

Multipart ETag that won't reconstruct

B2 (SHA1-native) and R2 (opaque) both break MD5-of-MD5s verification — a strong tell where Server is masked.

Hardest pair

AWS vs Wasabi. Wasabi is deliberately AWS-faithful — identical SigV4 debug fields and error shapes. Split them on the Server: WasabiS3/… string, the <CMReferenceId> error element, or the :A/:B request-id suffix.
